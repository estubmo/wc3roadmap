# Phase 9 — Guided Pathways & Launch: UI Design Contract

> Visual and interaction contract for Phase 9. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Sources: ADR 0001, `src/styles/app.css`, `components.json`, 09-CONTEXT.md (D-01–D-16, F-01 carry-forward),
> 02-UI-SPEC.md (established node/pathway visual language — extended, not replaced), `PathwayBanner.tsx`,
> `GraphNode.tsx`, `MasteryBadge.tsx`, `NodePanelContent.tsx`, `src/routes/__root.tsx`,
> REQUIREMENTS.md (PATH-01..04, CONT-04/05, PROG-05), ROADMAP.md §Phase 9 success criteria.

**This is an extension, not a fresh system.** Phase 9 ships zero new colors, zero new fonts, and reuses
the exact typography/spacing scale locked in Phase 2. Every new element below cites the existing
convention it extends.

**Primary focal point (main pathway view):** the "Next" node cue is the single strongest visual pull on
load — entrance animation `0.25s ease-out`, positioned directly above the first incomplete pathway step.
It draws the player's eye immediately after the intro overlay dismisses (first visit) or on landing
(returning visitor), answering "what do I do next?" before anything else on the canvas competes for
attention.

### Accent reserved for (extends the Phase 2 list — items 9–12 are new):

1–8. Unchanged, see `02-UI-SPEC.md §Color` (mastered/in-progress node encoding, edge highlight,
"Explore full map" CTA, text selection).
9. **Progress-bar fill** (`PathwayBanner`) — same semantic as "mastered," now expressed as a proportion
   fill rather than a node border. Lives in banner chrome, never on the node face — does not compete
   with the Phase 2 three-gold node hierarchy (F-01).
10. **Mastered step-number badge fill** (`rune-500` bg / `obsidian-950` text) — a small-scale echo of
    the existing "mastered" encoding (item 1), applied to the step-number chip only when that step's
    node is mastered. Not a new meaning — same token, same semantics, smaller surface.
11. **"Next" cue pill border + text** (`rune-600` border / `rune-400` text) — the *only* new semantic
    use of rune-gold on the node face this phase. Deliberately kept to border+text only (no fill, no
    glow) — the lowest-intensity treatment on the F-01 intensity scale, sitting below the edge-highlight
    (item 6) in visual weight so it reads as "guidance," not a fourth mastery state.
12. **Intro-overlay primary CTA** (`rune-500` fill / `obsidian-950` text) — reuses the exact
    `SignInButton` gold-CTA pattern (Phase 4) for consistency: this is the second "first-run gold CTA"
    in the product, and should look like the same affordance family.

**Staleness is intentionally NOT rune-gold** (new rule, item 13 exclusion): rune-gold means "skill
signal" everywhere else in the product (mastered / in-progress / next / CTA). Staleness is a trust
signal about *content*, not a mastery signal about the *player* — reusing gold for it would blur F-01's
hierarchy further and could misread as a positive marker. The staleness pill and canvas marker use
neutral `#e9e8ee` at reduced opacity instead — visually calm, legible, and semantically distinct.

## Pathway Completion Progress Bar (PATH-04, D-01/D-02/D-03)

**Location:** Replaces the existing static `"{N} of {total} nodes"` line in `PathwayBanner.tsx` (the
third line in the left identity column, currently 13px/400/opacity-0.5).

**Structure (top to bottom, replacing that one line):**
1. Slim fill bar: `height: 6px`, `border-radius: 9999px`, full width of the banner's left column
   (`max-width: 240px` so it doesn't stretch the whole banner on wide viewports).
   - Track: `background: var(--color-obsidian-700)`
   - Fill: `background: var(--color-rune-500)`, `width: {masteredCount / steps.length * 100}%`
   - Fill transitions via `motion/react`: `animate={{ width: ... }}`, `transition: { duration: 0.4, ease: "easeOut" }`
2. Label directly below the bar, 4px gap: **"{N} of {total} mastered"** — 13px/600/1.4, `color: inherit`
   at `opacity: 0.85` (brighter than the 0.5 it replaces — this is now a live status, not decorative
   metadata).

**Counting rule (D-02 — binding):** `masteredCount` = number of `pathway.steps` IDs whose
`masteryMap[id] === "mastered"`. `in-progress` nodes do **not** advance the bar — they keep their
existing gold-ring encoding on the canvas (Phase 2, unchanged) but contribute 0 to this count. Compute
via the existing `graph-store.ts` `masteryMap` — no new state.

**Completion state (D-03 — binding, no fanfare):** When `masteredCount === steps.length`:
- Bar fill: 100%, same `rune-500`, same style — no color change, no shimmer, no confetti.
- Label replaces with **"Fundamentals complete"** — 13px/600, `color: var(--color-rune-400)` (a subtle
  warmer tone than the 85%-opacity default, the only concession to "this matters," still text-only).
- No modal, no toast, no sound. This is the single required visual delta at 100% — anything more
  violates PROG-05 (no gamification) per D-03.

## Pathway Step Numbers + "Next" Cue (PATH-01/03, D-04)

**Step-number badge — every pathway node, all the time:**
- Position: `absolute`, top-left corner of the node face, offset `top: -8px; left: -8px` (overlapping
  the node border corner — same overlap technique already used for faction accent bars in `GraphNode.tsx`).
- Shape: 20px diameter circle, `border-radius: 9999px`.
- Numeral: the node's 1-based index in `pathway.steps[]` (D-04) — 10px/600, `font-family: var(--font-sans)`.
- Default styling (step not yet mastered): `background: var(--color-obsidian-800)`,
  `border: 1px solid var(--color-obsidian-600)`, text `#e9e8ee` at 80% opacity.
- Mastered styling (this step's node is mastered — accent item 10): `background: var(--color-rune-500)`,
  text `var(--color-obsidian-950)`, no border (matches the mastered node face's own border-radius logic).
- Only renders for nodes that are pathway steps. Non-pathway nodes never get a step-number badge (this
  is a pathway-only affordance, not a general graph feature).
- `aria-label`: `"Step {n} of {total}"` on the badge element.

**"Next" cue — exactly one node at a time (D-04):**
- Definition: the first `pathway.steps[]` entry whose `masteryMap[id] !== "mastered"`, in step order. If
  all steps are mastered, no node carries the cue (matches the D-03 completion state — nothing left to
  point at).
- Visual: a small pill positioned `absolute`, centered above the node, `top: -28px` (clears the
  step-number badge), containing the text **"Next"** (10px/600, uppercase, `letter-spacing: 0.05em`,
  `color: var(--color-rune-400)`) on `background: var(--color-obsidian-800)`,
  `border: 1px solid var(--color-rune-600)`, `border-radius: 9999px`, `padding: 2px 8px`.
- A small `ChevronDown` (lucide, 10px, `color: var(--color-rune-400)`) sits directly below the pill,
  pointing at the node.
- **No looping/continuous animation** — entrance only: `initial: { opacity: 0, y: -4 }`,
  `animate: { opacity: 1, y: 0 }`, `transition: { duration: 0.25, ease: "easeOut" }`, played once when
  the cue mounts on a node (including when it moves to a new node after the previous one is mastered).
  This keeps Phase 9 consistent with the Phase 2 rule ("no continuous loops") and avoids adding visual
  noise around an already gold-heavy node face.
- `aria-label="Next recommended step"` on the pill (screen-reader users get the same "what do I do next"
  signal without relying on position).
- **Does not appear together with a step-mastered badge on the same node** — by definition the "next"
  node is never mastered, so the step-number badge on that node always shows the default (non-gold)
  styling. No visual conflict between the two accent uses.

## First-Visit Intro Overlay (PATH-01/03, D-05)

Uses the existing shadcn `Dialog` component (`src/components/ui/dialog.tsx`, installed Phase 4) —
no new registry component.

| Element | Spec |
|---------|------|
| Trigger | Auto-opens on mount if `localStorage.getItem("wc3roadmap:pathway-intro-seen") !== "true"`; a first-time visitor only. |
| `DialogTitle` | "Start here" — 22px/600, Space Grotesk (uses the shadcn default `DialogTitle` styling, which already resolves to the app's `text-lg font-semibold` — override `className` to match the 22px/Space Grotesk pathway-heading role above for visual consistency with the banner it sits over) |
| `DialogDescription` / body | "8 fundamentals, ordered top to bottom. Click any node to learn — mastering them lights up your map." — 14px/400/1.65, Outfit (body-prose role) |
| Primary CTA | Button, full gold-CTA treatment (accent item 12): "Got it — let's start" |
| Dismiss | Clicking the CTA, the dialog's built-in `X` close, or pressing Escape all set the localStorage flag to `"true"` and close. No secondary "don't show again" checkbox needed — dismissal itself is the one-time flag (D-05: "gated by a localStorage flag so it appears once"). |
| Backdrop | Standard shadcn dialog overlay (`bg-black/50`) — clicking the backdrop also dismisses + sets the flag (Radix default `Dialog` behavior; do not override `onInteractOutside` to block it — this is not the `RegionSelector`'s captive-flow case). |
| Positioning | Centered overlay (shadcn default), sits above the pathway view which remains visible/dimmed underneath — the user sees the spotlighted pathway through the backdrop, reinforcing "this is what I'm about to look at." |

**localStorage key:** `wc3roadmap:pathway-intro-seen` (namespaced per the existing `wc3roadmap:` prefix
convention used elsewhere in local-progress storage — planner confirms exact existing prefix in
`src/lib/local-progress.ts` and matches it).

## Staleness Indicator — Node Detail Panel (D-06/D-07)

**Trigger (locked by CONTEXT D-06, not a design decision):**
`node.meta_volatile === true && CURRENT_PATCH.id !== node.patchId`.

**Location:** A slim strip directly below the panel header, above the scrollable body, in
`NodePanelContent.tsx` — replaces the deferred hook already flagged in that file (lines ~24/36/227).
Does not compete with the header row (title + mastery badge + close) which stays exactly as-is.

**Structure:**
- `padding: 6px 20px` (matches the panel's `20px` horizontal rhythm), `background: var(--color-obsidian-800)`,
  `border-bottom: 1px solid var(--color-obsidian-700)`.
- `Clock` icon (lucide, 12px, neutral `#e9e8ee` at 65% opacity) + 6px gap + text.
- Text: **"Unreviewed for {CURRENT_PATCH.label or .id}"** — 11px/600, same neutral tone (not rune-gold,
  per the Color section rationale above).
- The whole strip is a shadcn `Tooltip` trigger (`TooltipTrigger asChild`, wrapping the strip in a
  `<button>`-like element or using `tabIndex={0}` for keyboard/focus access — required for touch/mobile
  where hover doesn't exist, so also support tap-to-toggle if the planner determines Radix Tooltip
  doesn't cover touch by default).
- `TooltipContent`: **"Last reviewed for patch {node.patchId} — current patch is {CURRENT_PATCH.id}. This content may be out of date."** (exact copy below in the Copywriting Contract).
- Strip only renders when the trigger condition is true — absent entirely for reviewed/current-patch nodes (no layout shift placeholder needed elsewhere).

## Staleness Indicator — Graph Node Marker (D-08/D-09)

**Feeds from:** the new `stale: boolean` field on `GraphDisplayNode` (D-09 — computed in the `index.tsx`
loader from `patchId` + `meta_volatile` + `CURRENT_PATCH`, same trigger rule as the panel).

**Visual:** a small `Clock` icon (lucide, 10px), positioned `absolute`, `bottom: 4px; right: 4px` on the
node face (`GraphNode.tsx`) — the one previously-empty corner (top-left has the type icon, top-right has
mastery badge + source markers, bottom-left has difficulty dots).
- Color: `#e9e8ee` at 55% opacity — neutral, matching the panel pill (never rune-gold, per Color rationale).
- No background chip, no border — the icon alone, kept minimal so it doesn't compete with the difficulty
  dots on the same visual row.
- `aria-label="Content may be outdated for the current patch"` (`role="img"`), same pattern as the
  existing difficulty-dots/type-icon aria treatment in `GraphNode.tsx`.
- Renders only when `data.stale === true`. No marker, no reserved space, for non-stale nodes.

## Animation Contract

All animation via `motion/react` (never `framer-motion`). Respects `prefers-reduced-motion` (already
global in `app.css`).

| Element | Animation | Spec |
|---------|-----------|------|
| Progress bar fill | Width transition on mastery change | `animate={{ width }}`, `transition: { duration: 0.4, ease: "easeOut" }` |
| Step-number badge → mastered | Fill color transition | CSS `transition: background-color 0.3s ease, color 0.3s ease` (no Motion needed — simple color swap) |
| "Next" cue entrance | Fade + upward settle, plays once per node it lands on | `initial: { opacity: 0, y: -4 }`, `animate: { opacity: 1, y: 0 }`, `duration: 0.25, ease: "easeOut"` — no loop |
| Intro overlay open/close | shadcn `Dialog` default (`data-[state=open]:animate-in fade-in-0 zoom-in-95`, unchanged) | No override needed |
| Staleness elements | None — static, no entrance animation (informational, not celebratory or attention-grabbing) | — |
| "Fundamentals complete" label swap | Simple opacity crossfade | `AnimatePresence` + `initial: { opacity: 0 }` / `animate: { opacity: 1 }`, `duration: 0.2` |

No parallax, no scroll-linked animation, no continuous/looping motion anywhere in Phase 9 — consistent
with the Phase 2 baseline and reinforced by PROG-05 (no gamification flourish).

## Accessibility Baseline

| Concern | Contract |
|---------|----------|
| Progress bar | `role="progressbar"`, `aria-valuenow={masteredCount}`, `aria-valuemin={0}`, `aria-valuemax={total}`, `aria-label="Pathway completion"` on the track element |
| Step-number badges | `aria-label="Step {n} of {total}"` per badge (see above) |
| Next-node cue | `aria-label="Next recommended step"`; not conveyed by color alone — the text "Next" is always present |
| Intro overlay | Standard Radix `Dialog` focus trap + `Escape` to close (built-in); focus returns to the triggering context (the pathway view) on close |
| Staleness pill (panel) | Keyboard-focusable (`tabIndex={0}` if not a native interactive element), `Tooltip` content readable via focus, not hover-only — required for keyboard and touch users |
| Staleness graph marker | `role="img"` + `aria-label`, same pattern as existing difficulty dots — not interactive (no tooltip on canvas; full explanation lives in the panel) |
| Color contrast | Staleness neutral text (`#e9e8ee` at 55–65% opacity on `obsidian-800`/`obsidian-950`) must be verified ≥ 4.5:1 by the executor — if opacity drops contrast below threshold, increase to the nearest value that clears it before shipping |
| 404 / About pages | Standard heading hierarchy (`h1` → `h2`), focus starts at the page heading on route entry per existing router conventions |
| `prefers-reduced-motion` | Already global in `app.css` — the entrance-only animations above (progress fill, next-cue entrance) already collapse to instant per that rule; no extra work needed |
