| Boundary | Description |
|----------|-------------|
| localStorage payload → server merge | client-held progress is sent to mergeProgressOnSignIn across the trust boundary |
| session transition → one-time effect | merge must run exactly once, not on every load |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-07a | Tampering | merge payload | medium | mitigate | mergeProgressOnSignIn (05-04) validates every record with MasteryStateSchema and fill-gaps (server-wins) — poisoned local entries cannot overwrite server state |
| T-05-07b | Repudiation | repeated merge / data loss | medium | mitigate | wc3rm:merged flag + useRef guard ensure the merge runs once; server-wins means deliberate signed-in marks are never clobbered by stale local data (D-07) |
| T-05-07c | Info Disclosure | cross-user hydrate | high | mitigate | getUserProgress is principal-keyed (05-04); masteryMap only ever holds the authenticated principal's rows |
</threat_model>
